# Session Checkpoint — 2026-03-05-2108

## 1. Session Summary

This session completed all 4 planned v1 phases of Wingman v2 (browser terminal for Claude Code). Significant time was spent fixing bugs discovered during UAT.

### What was built:
- **Phase 3 (Mission Control)**: Dashboard UI at `/`, session terminals at `/session/:id`, REST API, WebSocket broadcast for real-time status, PID lock (single instance per dir), graceful shutdown, manual mode (`--manual`)
- **Phase 4 (Distribution)**: `bin/wingman.js` entry point, `package.json` bin/files/engines fields, Git Bash auto-detection across common Windows paths, auto-port (port 0, OS assigns free port so multiple instances in different dirs don't conflict)

### Bugs fixed this session:
- Reconnectable sessions not responding to keypresses (PTY was dead, needed `spawnAndWirePty` on reconnect)
- Multi-tab resize conflict causing both tabs to freeze (primary-WS pattern)
- Session descriptions not persisting across server restart (wasn't saved to `sessions.json`)
- Sessions showing as "closed" after restart instead of reopenable (changed `gracefulShutdown` to use `detachPty` not `closeSession`)
- Terminal letterbox (not filling full height) — CSS flex layout fix
- History replay garbling on second tab — deferred: raw PTY history with TUI escape sequences is imperfect
- "Close" button permanently killing sessions with no way back — redesigned to "Stop" (always restartable)
- "Wingman shutting down" overlay staying forever — now shows "Wingman stopped" on WS close
- MC not updating when `/exit` used in terminal — broadcast on WS disconnect

### Key design decisions made:
- Two session states only: **active** (green dot, Open+Stop) and **stopped** (grey dot, Open only). No permanent "closed" in UI.
- `Stop` = kill PTY but keep session restartable. Only way to permanently remove is direct deletion.
- Multiple Wingman instances in different dirs use auto-assigned ports — no conflicts.

## 2. Current State

**Branch:** `feature/wingman-v2`
**Latest commit:** `d4fc79d` — docs: update ROADMAP to reflect all 4 phases complete
**Ahead of origin by:** ~45 commits

### What's working:
- `node bin/wingman.js` (from testbed) opens Mission Control in browser
- New Session → prompts for name → opens terminal tab with Claude Code running
- Sessions named (not raw UUIDs), names persist across server restart
- Stop session → restartable, Open → fresh Claude in same named session
- Multiple instances in different directories simultaneously (different ports)
- Exit Wingman / Ctrl+C → clean shutdown, "Wingman stopped" shown in all tabs
- PID lock prevents duplicate instances in same directory

### What is NOT working / NOT built yet (CRITICAL GAP):
**The original Wingman v1 (Flutter app) maintained per-session prompt and context history.** Users could:
- Stage a prompt in the Wingman UI and inject it into Claude via `/ccp`
- Stage context in the Wingman UI and inject it via `/ccc`
- Browse previous prompts per session and reuse them

**None of this exists in v2.** The current v2 is purely a terminal viewer — there is no prompt/context management, no history browser, no staging area. This is the primary missing feature.

The v1 app wrote files to:
- `docs/cc_prompt.md` — active prompt
- `docs/cc_context.md` — active context
- `wingman/history/` — per-prompt JSON history files

Users then used shell aliases:
- `ccc` = cat the context file into Claude
- `ccp` = cat the prompt file into Claude

**In v2, the equivalent would be:**
- Each session has its own prompt and context area in the Mission Control / session UI
- User edits prompt/context in the browser
- Files written to `.ai/wingman/sessions/<id>/prompt.md` and `context.md`
- Claude Code slash commands `/ccc` and `/ccp` read from these files
- Full prompt history per session, browseable and reusable
